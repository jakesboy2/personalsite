Our Philosophy

  We hold certain development principles in high regard when it comes to developing our code. Constant refactoring, rethinking, and reimplmenting features in an attempt to make them cleaner, more effective, and less prone to errors is a constant battle that we seek to win.

  Maintainability is our top priority. After finishing a major feature we come together and discuss what can be improved. Often times, this means abstracting functionality into various design patterns, functions, or classes. Constant discussions and changes are considered and completed to restructure our AI in a way that will make it easier to work with based on our most common pain-points.

   We make use of various design patterns often within our development. There is also a high emphisis on principles meant to reduce coupling and increase coherence.

What Is Screeps?

  Screeps is a massive multiplayer online real-time strategy game. Each player can create their own colony in a single persistent world shared by all players.

  Using the game's provided API, you program your AI in Javascript, Typescript, or any language that compiles into web assembly.

  You control every aspect of the game from your code. Every 'tick' (roughly 2.5-3.5 seconds real-time), player's scripts are ran and intents are queued based on the results of your code. Intents include things such as calling a method on an instance of a Spawn object to create a new creep, moving a creep in a particular direction, upgrading your room, attacking another player, harvesting resources, and building or repairing various structures.

  Your code runs even when you are offline. The world is always running. Because of its nature as a sandbox game, there are no win-conditions or goals; however, there is a leaderboard that tracks the amount of energy you have used to upgrade your rooms. Claiming more rooms gives you more opportunity to gain control points, more territory in the world, and more notoriety amoung other players.

  There is a global market place you can use to trade resources with other players, and various powerful players have come together to create alliances which has resulted in large scale world wars breaking out.


Strategy Pattern

   We make use of this design pattern more than any other in our code-base. This was the result of our first major refactor. The issue that we were trying to solve was the amount of switch statements whose only purpose was to decide what helper function to call. Eventually, an idea was born. To demonstrate the concept and our application of it, I will use our API function that generates the spawning limits for each room.

   For a quick introduction to the context here, we have our room split up into multiple 'Room States' based on conditions being met such as, the amount of energy available, the presence of particular structures, and the presence of option flags. To replace the switch statement that decided how many of each creep we wanted at each room state, we needed a way to abstract out the entire concept of creep limits and have a file for each room state we want to configure the limits for.

Click the box on each step below to view the example.

  • Define Interface

    The first step is to set up an interface to use for generating the spawn limits for the room. This contains the definitions for generating domestic and remote limits, as well as a property reference to the name of the current room state. This is so we can find it at run-time.

  • Implementation Set-up

    We then defined a class that implements the interface from the previous step, for every room state we wanted to use. The only trick here is defining the name of the current room state as defined in class we are in, and binding the methods to the instance of this object.

  • Implementation Meat & Bones

    This is where we actual set up the specific room state's implementation. For this room state, we create and return the object containing all of the creep roles we want to define limits for. This same process is done below on the remote limits as well, as they are split into different methods for maintainability.

  • Array of Implementation Instances

    Here we simply define an array that contains the instance of each of these implementations. This is the array we will use to decide at run-time which instance we are going to call for each unique situation and state the code can be in at any given time.

  • Select an Implementation at Run-time

    For this step, we get the room state of the current room, and loop over the array created in the previous step until we find a match. From there we simply return the value of that implmentation's method. This allows each room to get its own custom object that defines the spawn limits for itself.

  • Setting the Limits

    In here, we use our Memory API to update the limits for the room, by passing in the room we are updating, and the object returned from that room's configured limits for this room state, generated using the above steps. The exception here is generating a military creep queue, because it works off a queue instead of limits; however, remote and domestic creeps are generated using the strategy pattern as described here.

   These steps are repeated for several modules of our code. Namely, creep roles, creep spawning helpers (options, name, target/home rooms, etc), flag handlers, and spawn limits. This design pattern provides a powerful tool for separating your logic and clearly seeing where each section of each state the room can be in is defined for easy maintainence, reliability, and bug-squashing.

Essentially, I find myself often writing a TODO comment to abstract something into a function, or improve the logic for a situation in an attempt to just get the working product out ASAP. What we attempt to do in this case is simply delete the comment, and do what we were writing the comment for.

The line lies in making sure you keep up with your TODO comments and issues as you finish features to make sure you don't collect a large backlog of incomplete features to build your future code off of.

   This also supports constant refactoring. We always look for chances to re-evaluate our code and find places to improve. This has resulted in a code base that has been in development now for over a year and has maintained clean and easy to add features and fix bugs within.

Creep Biography

Gus "Miner" Johnson

Before the Great War, Gus spent his days working in the energy mines with his father and four brothers. After an accident in the mines, Gus emerged as the sole survivor. With no family, no friends, and nowhere to go, Gus decided to take his talents to the new lands out west. He didn't know anything about it, just that his old friends called the foreign lands the 'Screeps', and complained they were going to put them out of business due to the abundance of energy sources and migrant workers ready to trade body for coin. Gus teammed up with some fresh migrants he met on the journey over and together they formed a community focused on surviving out in this new world. `

The miner creep is the foundation of the room. They sit diligantly on each energy source within the room and mine energy. They do not carry it, or move it, or place it anywhere, they simply mine it and allow it to drop below them, where a container will generally be in place to accept their offering. Without the miner, the room cannot exist.

Henry "Harvester" Harrington

Henry spent his days as an innkeeper of a five-star Yelp reviewed tavern. His tavern was often considered the greatest in all the lands until the Great War left it in burning ruins. Without a place to go, or a family to turn to, Henry searched for new lands to lay claim to. The Originals needed someone to take care of their day to day life, and Henry needed a place to call home.
The harvester creep is another integral part of the room. Part of the "Big Three" roles that are required to sustain a room. They handle moving energy from the miners to the storage, and then fill the spawn and extensions. They essentially handle keeping the spawn churning and provide energy for the workers.
Roy "Worker" Jefferson

Roy made a living doing odd jobs for everybody he knew. "Nobody in town was better with a tool" was a phrase that rang true no matter what town he happened to be in that week. Equipped with the skills to make anything happen, he was headhunted by the Originals seeking a handy-man to help strengthen their colony.
The worker is the final integral part of the "Big Three" roles required to run a room. They handle everything in the room that isn't logistics. This includes repairing, building, and upgrading the room itself at lower levels. They are only allowed to pull energy from the storage to use for the room, as to not starve the harvesters of their energy.
Daisy "Lorry" Neutron

Daisy spent many years in university learning the ins and outs of chemistry. On her last year, before graudation, the war broke out and left her university destroyed. Looking for somewhere to apply her skills, the Originals brought her on board as a full time scientist without hesitation to run their labs.
The Lorry handles all lab related opereations. They carry the resources from the storage into the labs, carry out the reactions and bring the finished product back to the storage.
Dave "Claimer" Crockett

Dave was one of the first Original creeps, and one of the first of any Creep, to explore the new territory. He spent years finding new lands to settle and decided to start finding others to build a colony with immediately following the Great War in interest of self preservation.
The Claimer travels to new rooms and claims them for the colony. Once claimed, their job has been completed and they are recycled.
Theadore "Mineral Miner" Bebola

Theadore spent his days searching for exotic minerals. He was never interested in being a mere miner for a large corporation, but rather find his own path and sell the exotic minerals he found for a pretty penny. Once the war had ended, Theadore needed a home, and the Originals needed minerals to supply the colony with the boosts needed for domination. It was a match made in heaven.
The Mineral Miner handles going to extractors and extracting minerals from it. They are self contained and move the minerals back to the storage on their own without the need of other creeps to get involved.
Derrick "Power Upgrader" Henry

Derrick was a football player before the war broke out. One of the greatest running backs in the league. Now that his money meant nothing, all he had was incredible size, strength, and tenacity. The Originals took him under their protection in exchange for his incredible strength.
The Power Upgrader sits on the controller and is supplied energy via Links to upgrade the controller full time. This frees up the time wasted traveling to the storage to get energy and allows the workers to focus on repairing the walls in the room, keeping the energy usage of the room more consistant and controllable.
Bill "Remote Colonizer" Lewis

Bill had extensive experience building settlements in the new world, considering how much money he made from doing so. After the war, it seemed the only settlments still standing were the ones designed and defended by him. The Originals noticed this, and quickly negogiated the integration of his skills into the colony.
The Remote Colonizer follows the claimer into the unknown, building up new rooms from scratch until the room is mature enough to handle itself.

Jessie "Remote Harvester" Yawnev

Jessie was the bartender at the same Inn as Henry Harrington, the Originals' main harvester. He wouldn't join unless she came along as well, as they make a wonderful team.
